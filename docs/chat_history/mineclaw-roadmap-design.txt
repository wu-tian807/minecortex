## MineClaw vs AgenticOS — 全面对比分析

两个框架虽然共享"目录即大脑"的哲学和 `while(tool_use)` 的 Agent Loop 范式，但定位截然不同：

| 维度 | MineClaw | AgenticOS |

### 五、核心结论

#### 1. Roadmap 实现后能否补全 AgenticOS 的所有 Agent 工具？

**不能完全补全，也不需要。** 原因有三：

- **定位不同**：MineClaw 是游戏 AI 框架，AgenticOS 是通用编程 Agent。约 40% 的 AgenticOS 工具（`read_file`, `write_file`, `edit_file`, `glob`, `grep`, `bash`, `lsp`, `notebook_edit`, `git_worktree`）对游戏 AI 无意义。
- **范式不同**：MineClaw 用"事件订阅"（stdin/heartbeat）替代了 AgenticOS 的"工具化交互"（`ask_user`/`sleep`）。用"对等多脑"替代了"Supervisor 层级"（`spawn_agent`/`agent_status`/`agent_kill`）。
- **未规划的缺口**：MCP 集成、Pipeline/Workflow 引擎、HTTP Channel 三块在 Roadmap 中完全未涉及。如果未来要变成通用框架，这些需要额外规划。

#### 2. Roadmap 实现后能否兼容 AgenticOS 的 Skills？

**结构上可以兼容，但执行层不完整。**

- MineClaw 的两层加载（摘要 + 按需读取）与 AgenticOS 的设计一致
- MineClaw 的 Skill 可以读取 AgenticOS 格式的 `SKILL.md`（都是 Markdown + 元数据）
- **但缺少 SkillExecutor**——AgenticOS 的 Skill 可以带 `scripts/run.ts` 脚本并执行，MineClaw 没有规划这个能力
- **缺少 `create_skill`**——运行时创建 Skill 的能力未工具化（虽然 evolve 模式理论上可以）

#### 3. Roadmap 实现后能否达到 AgenticOS 的对应功能？

**在核心 Agent 循环层面可以对齐，但在工具生态和平台能力上有显著差距。**

可以对齐的部分：
- Agent Loop 核心循环（已对齐）
- Session 管理 + 三层压缩（§1）
- 多 Agent 通信和协作（已有 BrainBus，§2 补充子任务）
- Prompt 组装的灵活性（§4 的 Slot 化甚至超越 AgenticOS）
- Skills 基础能力（§5）
- 权限系统（AGENTIC.md 已设计，更精细）

MineClaw 独有优势：
- **ScriptBrain**（§6）——AgenticOS 所有 agent 必须走 LLM，MineClaw 可以纯脚本（HTN/GOAP/BT），大幅降低 token 成本
- **Slot + Tool 两原语**——比 AgenticOS 的硬编码注入点更优雅、可扩展
- **对等多脑自治**——比 Supervisor 模式更灵活（每个脑有独立心跳、独立 model、独立订阅）
- **evolve 进化模式**——AI 可以创造新脑、修改自身，这是 AgenticOS 的 `--evolve` 模式的超集

无法对齐的差距：
- **工具生态**：30+ vs 2（即使 Roadmap 全部完成也只增加 ~5 个）
- **MCP 集成**：AgenticOS 有完整的 STDIO/SSE/HTTP 客户端 + 工具桥接
- **Pipeline/Workflow**：AgenticOS 有 `defineStep()` DSL + PipelineMemory + 确定性编排
- **通信渠道**：AgenticOS 有 CLI + HTTP REST API + SSE，MineClaw 只有 stdin
- **看门狗/进程管理**：AgenticOS 有 watchdog + ProcessRegistry，MineClaw 没有

## 关键洞察：MineClaw 的抽象层次更高

```
创建 brains/x/   = spawn_agent
读 state.json    = agent_status
```

MineClaw 的核心哲学是"目录即大脑"——所有状态都在文件系统中。
evolve 模式下脑可以修改自身（soul.md、brain.json、skills/、src/），
文件操作工具也会频繁读写 brains/ 目录。这些变更必须**实时生效**，不能靠重启。

### 需求来源

| 场景 | 变更的文件 | 期望行为 |
|------|----------|---------|
| ReflectBrain 修改 soul.md | `brains/reflect/soul.md` | 下次 tick slot:soul 自动用新内容 |
| 通过 edit_file 改 brain.json | `brains/social/brain.json` | Scheduler 感知配置变化，可能需要重启脑 |
| 创建新脑目录 | `brains/explorer/` | Scheduler 自动发现并启动 |
| 新增/修改 Skill | `skills/mine_diamond.md` | skill-loader 重新扫描，slot:skills 更新 |
| 新增/修改 Tool | `tools/navigate.ts` | tool-loader 重新扫描，脑下次 tick 可用新工具 |
| 新增/修改 Directive | `directives/safety.md` | directive-loader 重新扫描 |
| 修改 ScriptBrain 代码 | `brains/planner/src/index.ts` | ScriptBrain 重新加载脚本入口 |
| 删除脑目录 | `brains/explorer/` (rm -rf) | Scheduler 停止并注销该脑 |

### 核心设计

**单一 Watcher，事件分发**：一个 FSWatcher 监听整个项目根目录，
按路径前缀分发给不同的处理器。

```typescript
interface FSChangeEvent {
  type: "create" | "modify" | "delete";
  path: string;       // 相对项目根的路径
  isDir: boolean;
}

interface FSHandler {
  pattern: RegExp;     // 匹配的路径模式
  handle: (event: FSChangeEvent) => void | Promise<void>;
}
```

**处理器注册**：

```
brains/*/brain.json     → Scheduler.onBrainConfigChange()
brains/*/soul.md        → SlotRegistry 标记 slot:soul 需重载
brains/*/skills/*       → SkillLoader.invalidate(brainId)
brains/*/tools/*        → ToolLoader.invalidate(brainId)
brains/*/directives/*   → DirectiveLoader.invalidate(brainId)
brains/*/src/*          → ScriptBrain.reload(brainId)
brains/ (新目录)         → Scheduler.onBrainDiscovered(brainId)
brains/ (删目录)         → Scheduler.onBrainRemoved(brainId)
skills/*                → SkillLoader.invalidateGlobal()
tools/*                 → ToolLoader.invalidateGlobal()
directives/*            → DirectiveLoader.invalidateGlobal()
key/*                   → Provider 重新加载密钥（安全敏感，可选）
```

**防抖**：文件写入可能触发多次 change 事件（write + rename），
每个处理器 debounce 300ms，只处理最后一次。

**与 Slot 系统联动**（§4 完成后）：

文件变更 → Loader invalidate → 下次 assemblePrompt 时 Loader 重新填充对应 Slot。
不需要立即重新加载——**懒加载，tick 时生效**。
Slot 的 `content` 可以是 lazy getter（首次读取时加载，invalidate 后标记为 stale）。

### 需要做的

- **FSWatcher 类**：封装 `node:fs/watch`（recursive），路径过滤 + 防抖 + handler 分发
- **Scheduler 集成**：启动时创建 Watcher，注册 brain 目录变更 handler
- **Loader invalidate 协议**：各 Loader 暴露 `invalidate()` 方法，标记缓存失效
- **debounce 策略**：默认 300ms，brain.json 变更可延长到 1s（防误触）

### 涉及文件

- 新建 `src/core/fs-watcher.ts` — FSWatcher 封装（~80 行）
- `src/core/scheduler.ts` — 创建 Watcher + 注册 handler
- `src/loaders/*.ts` — 各 Loader 新增 `invalidate()` 方法

### 4.1 SlotRegistry — 核心数据结构

```typescript
interface ContextSlot {
  name: string;
  order: number;
  source: "file" | "loader" | "tool" | "framework";
  condition?: (ctx: SlotContext) => boolean;

  // --- 灵活内容 ---
  // content 可以是静态字符串，也可以是惰性求值函数（每次渲染时调用）。
  // 惰性求值实现按需加载：fs-watcher invalidate 后，下次渲染自动重新加载。
  content: string | (() => string | Promise<string>);

  // --- 可组合 ---
  // 复合 slot 由多个子条目组成，渲染时拼接。
  // 例如 slot:brain_board 由多条 TrackedTask 组成，
  // slot:skills 由多个 skill 摘要组成。
  // 单一内容的 slot 不需要 entries，直接用 content。
  entries?: SlotEntry[];

  // --- 生命周期 ---
  stale?: boolean;       // fs-watcher 标记为脏，下次渲染时重新求值 content
  ttl?: number;          // 自动过期（tick 数），过期后移除。用于 brain_board 的已完成任务
  createdAt?: number;    // 创建时间戳
}

interface SlotEntry {
  id: string;            // 条目唯一标识（如 tracked task ID、skill name）
  content: string;
  meta?: Record<string, unknown>;  // 任意元数据（状态、优先级等）
}
```

### 4.2 SlotRegistry API

```typescript
class SlotRegistry {
  // --- 基础 CRUD ---
  set(slot: ContextSlot): void;              // 注册或覆写整个 slot
  get(name: string): ContextSlot | undefined;
  delete(name: string): void;
  has(name: string): boolean;

  // --- 子条目操作（复合 slot）---
  addEntry(slotName: string, entry: SlotEntry): void;      // 添加子条目
  updateEntry(slotName: string, entryId: string, patch: Partial<SlotEntry>): void;
  removeEntry(slotName: string, entryId: string): void;
  getEntries(slotName: string): SlotEntry[];

  // --- 渲染 ---
  render(ctx: SlotContext): Promise<string>;  // filter → sort → resolve → join

  // --- 运行时动态 ---
  invalidate(name: string): void;            // 标记 stale，下次渲染重新求值
  invalidateByPattern(pattern: RegExp): void; // 批量 invalidate（如 /^directive:/)
  gc(): void;                                 // 清理过期 slot（ttl 到期的）
}
```

**渲染流程**：

```typescript
async render(ctx: SlotContext): Promise<string> {
  this.gc();  // 清理过期 slot

  const slots = this.all()
    .filter(slot => !slot.condition || slot.condition(ctx))
    .sort((a, b) => a.order - b.order);

  const parts: string[] = [];
  for (const slot of slots) {
    // 如果 stale 且 content 是函数，重新求值
    if (slot.stale && typeof slot.content === "function") {
      slot.content = await slot.content();
      slot.stale = false;
    }

    // 解析内容
    let text = typeof slot.content === "function"
      ? await slot.content()
      : slot.content;

    // 复合 slot：拼接 entries
    if (slot.entries?.length) {
      const entryTexts = slot.entries.map(e => e.content);
      text = text ? text + "\n" + entryTexts.join("\n") : entryTexts.join("\n");
    }

    if (text) parts.push(text);
  }

  return parts.join("\n\n");
}
```

context-engine 的 5 层硬编码全部消失，变成一次 `registry.render(ctx)`。

### 4.3 灵活性机制

#### 惰性求值 + fs-watcher 联动

Slot 的 content 可以是函数，实现按需加载：

```typescript
// soul.md 作为惰性 slot
registry.set({
  name: "soul",
  order: 0,
  source: "file",
  content: () => readFile(join(brainDir, "soul.md"), "utf-8"),
});

// fs-watcher 检测到 soul.md 变更时：
registry.invalidate("soul");
// → 下次 render() 时自动重新调用 content 函数，读到最新内容
```

不需要立即重载——**tick 时生效**，天然 debounce。

#### 复合 slot（子条目模式）

brain_board、skills 等 slot 是由多个独立条目组成的：

```typescript

```

3. **运行时动态注册**：工具通过 `ToolContext.writeSlot()` 在运行时创建/更新 slot。支持 `custom:*` 命名空间让工具给 LLM 留"便签"。支持 ttl 临时 slot。

这三个能力组合起来，Slot 系统就成了一个**可观测、可组合、运行时可编程的 system prompt 管理器**——任何需要"让 LLM 下次看到某些东西"的场景，都归结为写 Slot。

## AgenticOS 的 Heartbeat 机制

AgenticOS 不是纯粹的用户交互范式。它有三种触发源，通过 `TriggerQueue` 统一排队：

```typescript
type TriggerSource = 'user' | 'agent_message' | 'heartbeat';

const PRIORITY = {
  user: 0,          // 最高优先
  agent_message: 1, // 子 Agent 回传
  heartbeat: 2,     // 最低优先
};
```

### Heartbeat 做什么

`Heartbeat` 类每 60 秒 tick 一次，逻辑很简单：

```1:76:agentic_os/src/core/heartbeat.ts
// 核心 tick 逻辑
private tick(): void {
  if (!this.config.enableAutonomous) return;  // 可关闭
  if (!this.isHumanActive) return;            // 人走了就不跑

  const hasPendingMessages = this.messageBus.pending('team-lead') > 0;

  if (hasPendingMessages) {
    // 有子 Agent 消息待处理 → 以 agent_message 优先级触发
    this.triggerQueue.push('agent_message', 'Sub-agent messages pending');
  } else {
    // 无消息 → 以 heartbeat 低优先级触发
    this.triggerQueue.push('heartbeat', 'Periodic check-in');
  }
}
```

### 关键设计点

1. **人类活跃检测**：`humanIdleThresholdSec` 默认 3600s（1 小时）。如果用户超过 1 小时没有交互，heartbeat 停止触发——避免人走了 Agent 还在烧 token。

2. **智能触发抑制**：heartbeat 不是无脑唤醒。它检查 MessageBus 是否有待处理的子 Agent 消息。有消息就以更高优先级（`agent_message`）触发，没有才以最低优先级（`heartbeat`）触发。

3. **与 monitorLoop 配合**：`AgentSession.monitorLoop()` 是核心循环：

```typescript
while (!this.closed) {
  this.heartbeat.start();                    // 开始心跳
  const trigger = await triggerQueue.waitForNext();  // 阻塞等待
  this.heartbeat.stop();                     // 收到触发，停心跳

  if (trigger.source === 'agent_message') {
    // 子 Agent 消息 → 回调处理
  } else if (trigger.source === 'heartbeat') {
    // 心跳 → 回调处理
  } else if (trigger.source === 'user') {
    // 用户输入 → 回调处理
  }
```

## Coalesce（合并窗口）机制详解

### MineClaw 的实现

核心代码就在 `brain.ts` 的 `run()` 方法里，只有 3 行：

```56:79:mineclaw/src/core/brain.ts
  async run(signal: AbortSignal): Promise<void> {
    while (!signal.aborted) {
      // ① 阻塞等待：队列空时挂起，直到有非 silent 事件到来
      let trigger: Event;
      try {
        trigger = await this.eventQueue.waitForEvent(signal);
      } catch { break; }

      // ② 合并窗口：如果触发事件不是 immediate（priority > 0），等一小段时间
      if ((trigger.priority ?? 1) > 0 && this.coalesceMs > 0) {
        await sleep(this.coalesceMs);
      }

      // ③ 批量取出：把合并窗口内积累的所有事件一次性拿走
      const events = this.eventQueue.drain();
      if (events.length === 0) continue;

      // ④ 处理整批
      await this.process(events);
    }
  }
```

**时序图**：

```
时间轴 →

事件到达:  [stdin消息]         [heartbeat]  [bus消息]
           ↓                    ↓            ↓
           t=0                  t=100ms      t=200ms

脑的行为:  waitForEvent 唤醒
           ↓
           priority=1 > 0 → sleep(300ms)
           ↓
           t=300ms: drain() → 拿到 3 个事件
           ↓
           process([stdin消息, heartbeat, bus消息])  ← 一次 LLM 调用处理全部
```

**关键行为**：

1. **priority 0 跳过合并**：`(trigger.priority ?? 1) > 0` 这个条件意味着 `priority: 0`（immediate）的事件**不等待**，立即 drain + process。典型场景：`send_message` 发的跨脑消息是 priority 0，收到后立即处理。

2. **合并窗口期间新来的事件自动入队**：`sleep(300ms)` 期间 EventQueue 的 `push()` 仍在工作，后来的事件默默入队。300ms 后 `drain()` 一次性全拿走。

3. **drain 按 priority 排序**：拿出来后先排序再交给 LLM。priority 0 排前面，heartbeat（priority 2）排最后。LLM 自然先处理重要的。

### 三方对比

| | AgenticOS | openclaw | MineClaw |
|---|---|---|---|
| **有 coalesce 吗** | **没有** | **有，250ms 默认** | **有，300ms 默认** |
| **实现方式** | TriggerQueue 取一个处理一个 | `setTimeout` + 合并窗口 + pending Map | `sleep(coalesceMs)` + `drain()` |
| **复杂度** | 最简单（~60 行） | 最复杂（~260 行），含 retry/generation/targeted wake | 极简（3 行逻辑） |
| **priority 跳过** | 有（user > agent_message > heartbeat） | 有（`coalesceMs: 0` 立即触发） | 有（priority 0 跳过 sleep） |
| **批处理** | 不是——每次只取一个 Trigger | 是——`pendingWakes` Map 批量处理 | 是——`drain()` 全部取出 |
| **去重** | 不去重 | 同 target 只保留最高优先级 | 不去重（但批处理+排序自然降噪） |

### AgenticOS 为什么不需要 coalesce

AgenticOS 是**用户对话驱动**的。它的 `TriggerQueue.waitForNext()` 每次只取一个 Trigger：

```typescript
async waitForNext(): Promise<Trigger> {
  if (this.queue.length > 0) return this.queue.shift()!;
  return new Promise(resolve => { this.resolver = resolve; });
}
```

取一个，跑一轮完整的 Agent Loop（可能几十秒），跑完再取下一个。不存在"短时间内大量事件涌入"的场景——用户打字没那么快，heartbeat 60 秒才一次。

而且 AgenticOS 的 heartbeat 本身就有**智能抑制**——它检查 MessageBus 有没有 pending 消息，没有的话 heartbeat 事件在 UI 层直接被忽略（不触发 `runTurn`）。所以 AgenticOS 把"该不该处理"的判断放在了**push 侧**和**消费侧**，而非用合并窗口。

### openclaw 的 coalesce 为什么这么复杂

openclaw 是**多 Agent 服务端框架**（gateway 模式），有很多并发的信号源：

- 多个 Agent 各自有 heartbeat interval
- state.json 文件变更（chokidar 监听）触发即时 heartbeat
- 重启请求可能连续触发多次
- 每个 Agent 有独立的 wake target

所以 openclaw 的 coalesce 需要：
- **per-target 去重**：`pendingWakes` 是一个 Map，key 是 `agentId::sessionKey`。同一个 Agent 的多次 wake 请求只保留最高优先级的那个
- **timer 抢占**：新请求比已排定的 timer 更紧急时，取消旧 timer 换上新的
- **retry 保护**：retry 类型的 timer 不可被普通请求抢占（防止退避被破坏）
- **generation 防腐**：handler 换代后旧的 disposer 不会误清新 handler

### MineClaw 的设计来源

MineClaw 的 coalesce 不是照搬哪一个，而是**取了 openclaw 的思想 + 极简化实现**：

- **思想来自 openclaw**：合并窗口 + 批处理 + priority 跳过
- **实现参考 AgenticOS 的简洁性**：不搞 Map/timer/generation，一个 `sleep` 搞定

这是因为 MineClaw 的场景处于两者之间——它是事件驱动的（不像 AgenticOS 那么闲），但每个脑是独立循环（不像 openclaw 那样有 gateway 级并发）。所以 `sleep(coalesceMs) + drain()` 刚好够用：

```
openclaw 的复杂度：处理多 Agent × 多信号源 × 服务端并发
MineClaw 的复杂度：单脑 × 多事件源 × 进程内
AgenticOS 的复杂度：单 Agent × 用户输入 × 偶尔 heartbeat
```

MineClaw 的 3 行 coalesce 在它的复杂度级别上是最优解——足够处理事件合并，又不引入 openclaw 那套 timer 管理的复杂度。

### 4.1 SlotRegistry — 核心数据结构

```typescript
interface ContextSlot {
  name: string;
  order: number;
  source: "file" | "loader" | "tool" | "framework";
  condition?: (ctx: SlotContext) => boolean;

  // --- 灵活内容 ---
  // content 可以是静态字符串，也可以是惰性求值函数（每次渲染时调用）。
  // 惰性求值实现按需加载：fs-watcher invalidate 后，下次渲染自动重新加载。
  content: string | (() => string | Promise<string>);

  // --- 可组合 ---
  // 复合 slot 由多个子条目组成，渲染时拼接。
  // 例如 slot:slot_board 由多条 TrackedTask 组成，
  // slot:skills 由多个 skill 摘要组成。
  // 单一内容的 slot 不需要 entries，直接用 content。
  entries?: SlotEntry[];

  // --- 生命周期 ---
  stale?: boolean;       // fs-watcher 标记为脏，下次渲染时重新求值 content
  ttl?: number;          // 自动过期（tick 数），过期后移除。用于 slot_board 的已完成任务
  createdAt?: number;    // 创建时间戳
}

interface SlotEntry {
  id: string;            // 条目唯一标识（如 tracked task ID、skill name）
  content: string;
  meta?: Record<string, unknown>;  // 任意元数据（状态、优先级等）
}
```

### 4.2 SlotRegistry API

```typescript
class SlotRegistry {
  // --- 基础 CRUD ---
  set(slot: ContextSlot): void;              // 注册或覆写整个 slot
  get(name: string): ContextSlot | undefined;
  delete(name: string): void;
  has(name: string): boolean;

  // --- 子条目操作（复合 slot）---
  addEntry(slotName: string, entry: SlotEntry): void;      // 添加子条目
  updateEntry(slotName: string, entryId: string, patch: Partial<SlotEntry>): void;
  removeEntry(slotName: string, entryId: string): void;
  getEntries(slotName: string): SlotEntry[];

  // --- 渲染 ---
  render(ctx: SlotContext): Promise<string>;  // filter → sort → resolve → join

  // --- 运行时动态 ---
  invalidate(name: string): void;            // 标记 stale，下次渲染重新求值
  invalidateByPattern(pattern: RegExp): void; // 批量 invalidate（如 /^directive:/)
  gc(): void;                                 // 清理过期 slot（ttl 到期的）
}
```

**渲染流程**：

```typescript
async render(ctx: SlotContext): Promise<string> {
  this.gc();  // 清理过期 slot

  const slots = this.all()
    .filter(slot => !slot.condition || slot.condition(ctx))
    .sort((a, b) => a.order - b.order);

  const parts: string[] = [];
  for (const slot of slots) {
    // 如果 stale 且 content 是函数，重新求值
    if (slot.stale && typeof slot.content === "function") {
      slot.content = await slot.content();
      slot.stale = false;
    }

    // 解析内容
    let text = typeof slot.content === "function"
      ? await slot.content()
      : slot.content;

    // 复合 slot：拼接 entries
    if (slot.entries?.length) {
      const entryTexts = slot.entries.map(e => e.content);
      text = text ? text + "\n" + entryTexts.join("\n") : entryTexts.join("\n");
    }

    if (text) parts.push(text);
  }

  return parts.join("\n\n");
}
```

context-engine 的 5 层硬编码全部消失，变成一次 `registry.render(ctx)`。

### 4.5 与 fs-watcher（§0）联动

```
文件变更 → FSWatcher 检测
  → 按路径匹配 Handler
  → Handler 调用 registry.invalidate("对应slot名")
  → Slot 标记为 stale
  → 下次 brain.process() 调用 registry.render()
  → stale 的 slot 重新调用 content 函数
  → LLM 看到最新内容
```

全程无阻塞，无主动重载——**所有热更新都是 tick 驱动的惰性生效**。

  new_string: ## 4. ContextSlot 化 — 统一的 LLM 上下文管理

当前 `context-engine.ts` 用硬编码拼接 5 层 system prompt，不可扩展。
需要一个统一的抽象来管理 LLM 能看到的一切——system prompt 和 conversation messages 都包含在内。

### 核心设计：两类 Slot

Slot 是 LLM 上下文中的一个**命名区域**。所有 LLM 能看到的内容都是 Slot，分为两类：

**固定槽（System Slot）**：渲染到 system prompt，跨 tick 持久存在。

```
[固定槽] "soul"              order: 0     ← soul.md
[固定槽] "runtime"           order: 10    ← 框架自动生成
[固定槽] "slot_board"        order: 20    ← 派发任务时自动订阅
[固定槽] "tools"             order: 30    ← tool-loader
[固定槽] "directive:*"       order: 40+   ← 指令模块（条件加载）
[固定槽] "skills"            order: 50    ← skill 摘要
[固定槽] "todos"             order: 60    ← 工具可写
[固定槽] "world"             order: 70    ← MineAvatar 快照
```

**增量槽（Message Slot）**：渲染到 conversation messages，本 tick 用完清空。
增量槽对齐 LLM API 的 message 结构，分三种 role：

| 增量槽 role | 对应 LLM message | 约束 | 典型内容 |
|------------|-----------------|------|---------|
| `user` | `{ role: "user" }` | 自由 | stdin 消息、BrainBus 收到的消息、heartbeat 信号 |
| `assistant` | `{ role: "assistant" }` | 后面必须跟 user 或 tool | LLM 上一轮的回复（session history 管理） |
| `tool` | `{ role: "tool", toolCallId }` | 必须紧跟对应的 assistant tool_call | 工具执行结果 |

`tool` 槽是 `assistant` 槽的附属——模型厂商要求 tool message 必须跟在对应的 assistant tool_call 后面，
不能独立存在。这个约束由 SlotRegistry 的渲染逻辑保证。

### 为什么这样分

```
LLM API 要求的 message 结构：
┌─────────────────────────────────────────────┐
│ system: "..."                               │ ← 固定槽渲染
│                                             │
│ messages: [                                 │ ← 增量槽渲染 + session history
│   { role: "user", content: "..." },         │    ← user 增量槽
│   { role: "assistant", content: "...",       │    ← session history
│     tool_calls: [...] },                    │
│   { role: "tool", content: "...",           │    ← tool 增量槽（附属于上面的 assistant）
│     tool_call_id: "..." },                  │
│   { role: "assistant", content: "..." },    │    ← session history
│   { role: "user", content: "..." },         │    ← user 增量槽（本 tick 事件）
│ ]                                           │
└─────────────────────────────────────────────┘
```

固定槽管 system prompt（"你是谁、你能做什么、你在追踪什么"）。
增量槽管 conversation（"刚才发生了什么"），必须遵守 LLM API 的 role 顺序约束。

### 4.1 类型定义

```typescript
type SlotKind = "system" | "user" | "assistant" | "tool";

interface ContextSlot {
  name: string;
  kind: SlotKind;
  order: number;                   // 同 kind 内的排序
  source: "file" | "loader" | "tool" | "framework" | "event-router";
  condition?: (ctx: SlotContext) => boolean;

  content: string | (() => string | Promise<string>);
  entries?: SlotEntry[];           // 复合 slot（slot_board、skills 等）

  // 生命周期
  ephemeral?: boolean;             // true = 渲染后清空（增量槽默认 true）
  stale?: boolean;                 // fs-watcher 标记脏
  ttl?: number;                    // tick 数后过期
}

interface SlotEntry {
  id: string;
  content: string;
  meta?: Record<string, unknown>;
}

// tool 增量槽的额外约束
interface ToolMessageSlot extends ContextSlot {
  kind: "tool";
  toolCallId: string;              // 必须关联到对应的 assistant tool_call
}
```

固定槽（`kind: "system"`）默认 `ephemeral: false`，跨 tick 持久。
增量槽（`kind: "user" | "assistant" | "tool"`）默认 `ephemeral: true`，渲染后清空。

### 4.2 SlotRegistry API

```typescript
class SlotRegistry {
  // --- 基础 CRUD ---
  set(slot: ContextSlot): void;
  get(name: string): ContextSlot | undefined;
  delete(name: string): void;
  has(name: string): boolean;

  // --- 子条目操作（复合 slot）---
  addEntry(slotName: string, entry: SlotEntry): void;
  updateEntry(slotName: string, entryId: string, patch: Partial<SlotEntry>): void;
  removeEntry(slotName: string, entryId: string): void;

  // --- 渲染 ---
  renderSystem(ctx: SlotContext): Promise<string>;         // 固定槽 → system prompt
  renderMessages(ctx: SlotContext): Promise<LLMMessage[]>;  // 增量槽 → message 数组
  flush(): void;                                            // 清空 ephemeral slots

  // --- 运行时 ---
  invalidate(name: string): void;
  invalidateByPattern(pattern: RegExp): void;
  gc(): void;
}
```

**两个渲染入口**，对应 LLM API 的两个部分：

```typescript
// renderSystem: 固定槽 → system prompt 字符串
async renderSystem(ctx: SlotContext): Promise<string> {
  this.gc();
  const systemSlots = this.allByKind("system")
    .filter(s => !s.condition || s.condition(ctx))
    .sort((a, b) => a.order - b.order);

  const parts: string[] = [];
  for (const slot of systemSlots) {
    const text = await this.resolveContent(slot);
    if (text) parts.push(text);
  }
  return parts.join("\n\n");
}

// renderMessages: 增量槽 → LLMMessage[]
async renderMessages(ctx: SlotContext): Promise<LLMMessage[]> {
  const messages: LLMMessage[] = [];

  // user 增量槽：合并为一条 user message
  const userSlots = this.allByKind("user")
    .filter(s => !s.condition || s.condition(ctx))
    .sort((a, b) => a.order - b.order);

  if (userSlots.length > 0) {
    const parts: string[] = [];
    for (const slot of userSlots) {
      const text = await this.resolveContent(slot);
      if (text) parts.push(text);
    }
    messages.push({ role: "user", content: parts.join("\n\n") });
  }

  // tool 增量槽保持独立（每个 toolCallId 一条 message）
  // assistant 增量槽通常由 session history 管理，不在这里渲染

  return messages;
}

// flush: 清空所有 ephemeral slot
flush(): void {
  for (const slot of this.all()) {
    if (slot.ephemeral) {
      if (slot.entries) slot.entries = [];
      else this.delete(slot.name);
    }
  }
}
```

### 4.3 Event Router — 事件到 Slot 的桥梁

EventQueue 是调度层（唤醒 + 合并 + 排序），drain 出来的事件交给 Event Router，
Router 决定每个事件写入哪些 Slot：

```typescript
function routeEvents(events: Event[], registry: SlotRegistry): void {
  for (const event of events) {
    // 所有事件 → user 增量槽（按 source 分组）
    const slotName = `events:${event.source}`;
    if (!registry.has(slotName)) {
      registry.set({
        name: slotName,
        kind: "user",
        order: sourceOrder(event.source),
        source: "event-router",
        ephemeral: true,
        entries: [],
      });
    }
    registry.addEntry(slotName, {
      id: `${event.source}_${event.ts}`,
      content: `[${event.type}] ${renderEventDisplay(event)}`,
    });

    // 特定事件 → 固定槽更新（副作用）
    if (event.source === "bus" && event.payload?.taskId) {
      registry.updateEntry("slot_board", event.payload.taskId, {
        content: `[✓] ${event.payload.summary}`,
        meta: { status: "completed" },
      });
    }
    if (event.source === "world" && event.type === "snapshot") {
      registry.set({
        name: "world", kind: "system", order: 70,
        source: "framework", content: event.payload.snapshot,
      });
    }
  }
}
```

**三层职责**：

```
EventQueue（调度层）
  push → waitForEvent → sleep(coalesce) → drain → Event[]
  不关心内容和渲染

Event Router（路由层）
  Event[] → 写入对应 Slot
  决定"这个事件该进哪些槽"

SlotRegistry（渲染层）
  renderSystem() → system prompt
  renderMessages() → LLMMessage[]
  flush() → 清空增量槽
  不关心事件从哪来
```

### 4.4 brain.ts 中的完整流程

```typescript
async process(events: Event[]) {
  // ① 事件路由到 Slot
  routeEvents(events, this.slotRegistry);

  // ② 渲染固定槽 → system prompt
  const systemPrompt = await this.slotRegistry.renderSystem(ctx);

  // ③ 渲染增量槽 → 本 tick 的 message
  const newMessages = await this.slotRegistry.renderMessages(ctx);

  // ④ 组装完整 messages
  const messages: LLMMessage[] = [
    { role: "system", content: systemPrompt },
    ...this.sessionHistory,
    ...newMessages,
  ];

  // ⑤ LLM 推理 + tool loop
  // （tool loop 中产生的 assistant/tool messages 由 brain.ts 直接管理，
  //   不走 SlotRegistry——它们是实时的对话流，不是 Slot）
  // ...

  // ⑥ 清空增量槽（本 tick 的事件已处理）
  this.slotRegistry.flush();
}
```

### 关键发现：没有人用 action/subcommand 模式

两个框架 **都没有** 采用 `tool(action: "read" | "write" | "delete", ...)` 这种合并模式。原因总结如下：

**1. 对 LLM 更友好**

细粒度工具让模型的 **tool_choice 决策更简单**。模型只需选「用哪个工具」，而不是先选工具再选 action。减少一层决策 = 减少出错概率。

**2. 参数 schema 更清晰**

`read_file` 只需要 `file_path`，而 `write_file` 需要 `file_path` + `content`。如果合成一个工具，参数变成条件必填（action=read 时 content 可选，action=write 时 content 必填），LLM 容易搞混。

**3. 权限控制天然细粒度**

Claude Code 能做到「这个命令只允许 Read + Grep」而禁止 Write，如果合并成一个 `file_tool`，就无法细粒度限权。Gemini CLI 也支持 `excludeTools` 排除特定工具。

**4. 工具描述更聚焦**

每个工具的 description 可以写得很精确，LLM 更容易理解何时该用。合并后 description 会变长、变模糊。

### 但也有 「适度合并」 的例子

两个框架并非完全不合并，有一些微妙的合并点：

| 合并场景 | 框架 | 做法 |
|---------|------|------|
| 单文件读取 vs 批量读取 | Gemini | 拆成 `read_file` + `read_many_files` |
| 单处编辑 vs 多处编辑 | Claude | 拆成 `Edit` + `MultiEdit` |
| Bash 命令过滤 | Claude | `Bash` 一个工具，通过命令模式 `Bash(git:*)` 控制 |
| 网页抓取的两种模式 | Gemini | `web_fetch` 一个工具，参数区分标准/实验模式 |

可见：**当操作的核心语义相同、只是参数维度不同时**，可以考虑合并；但 **当操作语义不同（读 vs 写 vs 删）时，一定拆开**。

```typescript
interface BrainAwarePath {
  path: string;        // 相对路径
  brain?: string;      // brain id，省略时为调用者自身
}

function resolvePath(input: BrainAwarePath, ctx: ToolContext): string {
  if (input.brain) {
    return join(BRAINS_DIR, input.brain, input.path);
  }
  // 无 brain 参数：如果路径看起来是 brain 内部文件（soul.md、brain.json、
  // skills/、memory/ 等已知 pattern），默认解析到调用者自身的 brain 目录
  if (isBrainLocalPattern(input.path)) {
    return join(BRAINS_DIR, ctx.selfId, input.path);
  }
  // 否则按项目根目录解析
  return join(PROJECT_ROOT, input.path);
}
```
